### Decisões de escopo desta sub-fase (brainstorm)

Durante o brainstorm, **comparação de períodos** e **toggle de granularidade
Dia/Semana/Mês** foram **removidos do escopo** — e do projeto. Motivo honesto:

- O cache guarda *fotos do agora*, não histórico. `fato_estoque_saldo` é
  "saldo hoje"; não existe "saldo de ontem" armazenado. Comparar saldo com
  período anterior é impossível com o cache atual e semanticamente sem sentido.
- `fato_estoque_movimento` é construído pelo worker **já agregado por mês**.
  Um toggle Dia/Semana exigiria reconstruir o fato em grão de dia — mudança de
  worker, fora do escopo de uma sub-fase de frontend.

Resultado: F3.5b entrega só o que o dado honestamente suporta — um seletor de
período **simples**, em grão de mês, sem tocar no worker nem nos fatos.

## Objetivo

Substituir o filtro de período atual (`PeriodFilter`, grão de mês, cru) por uma
barra de período moderna — pílulas de presets + calendário personalizado — nos
relatórios que têm dimensão temporal, com todo o estado na URL.

Hoje, apenas o relatório **"Entradas vs. saídas por mês"** (`entradas-saidas`)
tem dimensão temporal. Relatórios de snapshot (saldo, valor, parados, top
movimentados, concentração) **não** exibem a barra.

## Não-objetivos

- Comparação de períodos (removido — sem fundação de dados).
- Toggle de granularidade Dia/Semana/Mês (removido — exigiria mudança de fato).
- `KPICard` com delta ↑↓ (não nasce — dependia da comparação).
- Qualquer mudança no worker, nos builders de fato ou no schema Prisma.
- Granularidade abaixo de mês.

## Modelo de dados de período

Grão de **mês**. Um período é um par `{ de, ate }` de meses no formato
`YYYY-MM`, inclusivo nas duas pontas.

### Presets

| Preset (`periodo`) | Rótulo | Resolução |
|---|---|---|
| `mes` | Este mês | `de = ate = mês corrente` |
| `3meses` | Últimos 3 meses | `ate = mês corrente`, `de = mês corrente − 2` |
| `ano` | Este ano | `de = janeiro do ano corrente`, `ate = mês corrente` |
| `tudo` | Tudo | `de`/`ate` indefinidos → query não filtra por período |
| `custom` | Personalizado | `de`/`ate` vêm dos parâmetros `de` e `ate` da URL |

Preset padrão quando a URL não traz `periodo`: declarado no catálogo
(`temporal.periodoPadrao`). Para `entradas-saidas`: `3meses`.

## Estado na URL

A `searchParams` é a **única fonte de verdade**. Deep-link e botão "voltar"
funcionam sem estado client-side paralelo.

| Param | Valores | Quando presente |
|---|---|---|
| `periodo` | `mes` \| `3meses` \| `ano` \| `tudo` \| `custom` | sempre que ≠ padrão |
| `de` | `YYYY-MM` | apenas quando `periodo=custom` |
| `ate` | `YYYY-MM` | apenas quando `periodo=custom` |

Regras:
- `periodo` ausente → usa `temporal.periodoPadrao` do catálogo.
- `periodo=custom` sem `de`/`ate` válidos → cai para o preset padrão.
- `de > ate` → as duas pontas são trocadas (normalização silenciosa).
- Valor de `periodo` desconhecido → preset padrão.

## Camada de período — `src/lib/reports/periodo.ts`

Módulo de lógica **pura** (sem React, sem Prisma), 100% testável isolado.

```ts
export type PeriodoPreset = "mes" | "3meses" | "ano" | "tudo" | "custom";

/** Par de meses inclusivo; de/ate ausentes = sem limite (preset "tudo"). */
export interface PeriodoResolvido {
  preset: PeriodoPreset;
  de: string | null;   // "YYYY-MM" ou null
  ate: string | null;  // "YYYY-MM" ou null
}

/** Resolve searchParams + padrão do catálogo num período concreto. */
export function resolverPeriodo(
  params: { periodo?: string; de?: string; ate?: string },
  padrao: PeriodoPreset,
  hoje?: Date,
): PeriodoResolvido;

/** Serializa um período de volta para um objeto de searchParams. */
export function periodoParaParams(p: PeriodoResolvido): Record<string, string>;

/** Valida o formato YYYY-MM. */
export function ehMesValido(s: string): boolean;
```

`resolverPeriodo` aceita um `hoje` opcional para os testes fixarem a data.

## Catálogo

`ReportEntry` (`src/lib/reports/types.ts`) ganha um campo opcional:

```ts
/** Declara que o relatório tem dimensão temporal. Ausente = snapshot. */
temporal?: {
  periodoPadrao: PeriodoPreset;
};
```

- `entradas-saidas` passa a declarar `temporal: { periodoPadrao: "3meses" }`.
- O filtro `{ tipo: "periodo" }` é **removido** das `secoes[].filtros` desse
  relatório — o período deixa de ser um filtro de seção e vira dimensão do
  relatório.
- `ReportFilterTipo` perde o membro `"periodo"`.

## Componentes

Todos seguem o `ui-ux-pro-max` (tokens semânticos, foco visível, alvo de
toque ≥ 44px, `cursor-pointer`, `prefers-reduced-motion`).

### `PeriodBar` — `src/components/reports/period-bar.tsx`

- Client component. Renderiza a fileira de pílulas + o gatilho "Personalizado".
- Rótulo "PERÍODO" em `text-xs uppercase tracking-wide text-muted-foreground`.
- Pílula ativa: fundo violet (`bg-primary text-primary-foreground`); inativas:
  `bg-muted hover:bg-muted/70`. Transição 150–200ms.
- "Personalizado": pílula com ícone `CalendarDays` (lucide) que abre o
  `CustomRangePopover`. Quando o período é `custom`, exibe o intervalo escolhido
  no rótulo (ex.: "jan/2026 – mar/2026").
- Ao escolher um preset/intervalo, faz `router.push` atualizando os params
  via o helper de URL existente (mesmo padrão do `ReportFilters`).
- `role="group"` com `aria-label="Período"`; `aria-pressed` nas pílulas.

### `CustomRangePopover` — `src/components/reports/custom-range-popover.tsx`

- Popover (base-ui) com seleção de **mês inicial** e **mês final**.
- Implementação: dois seletores mês/ano (não calendário de dias — o grão é
  mês). Navegação por ano com chevrons; grade de 12 meses.
- Botões `Cancelar` e `Aplicar`; "Aplicar" só habilita com intervalo válido.
- Texto-guia da data-limite mínima: o catálogo **não** declara limite nesta
  v1 — a UI permite qualquer mês ≤ mês corrente; meses futuros ficam
  desabilitados.
- Fecha no `Esc` e no clique fora; foco volta ao gatilho.

### Integração — `report-view.tsx` / `page.tsx`

- `page.tsx` (server) resolve o período com `resolverPeriodo` e passa o
  `PeriodoResolvido` para as queries e para o `ReportView`.
- `ReportView` renderiza a `PeriodBar` no topo **apenas** quando
  `report.temporal` existe.
- A `PeriodBar` substitui o `PeriodFilter` na barra de filtros; os demais
  filtros (`armazem` etc.) continuam no `ReportFilters`.

## Queries

`ReportFilterValues` (`types.ts`): os campos `periodoDe`/`periodoAte` passam a
ser alimentados pelo `PeriodoResolvido` resolvido em `page.tsx`, não mais pelo
parser de filtro de seção. O formato continua `YYYY-MM`.

`getRelatorioEntradasSaidas` (`src/lib/actions/report-data.ts`): nenhuma
mudança de assinatura — já recebe `periodoDe`/`periodoAte`. Quando ambos forem
`null` (preset `tudo`), a query não aplica recorte de período.

`parseFilters` (`src/lib/reports/filters.ts`): deixa de tratar o tipo
`periodo`; o período passa a ser injetado por `page.tsx`.

## Estados e erros

- **Sem dados no período**: o `ChartEmpty` (já existente, F3.5a) cobre — a
  `LineChart`/`BarChart` mostra o empty state com a dica de ajustar o período.
- **Período inválido na URL**: normalizado silenciosamente pela camada
  `periodo.ts` (ver regras acima) — nunca quebra a página.
- **Preset `tudo`**: a query roda sem filtro de período; pode trazer muitos
  meses — aceitável para o volume atual (`fato_estoque_movimento` ≈ 12k linhas,
  poucas dezenas de meses agregados).
