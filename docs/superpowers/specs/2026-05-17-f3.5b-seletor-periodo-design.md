Decisão de brainstorm: o escopo é grande demais para uma spec única e foi

| F3.5f — Imersão (insights automáticos, atalhos de teclado, tour) | futura |

### Decisões de escopo desta sub-fase (brainstorm)

Durante o brainstorm, **comparação de períodos** e **toggle de granularidade
Dia/Semana/Mês** foram **removidos do escopo** — e do projeto. Motivo honesto:

- O cache guarda *fotos do agora*, não histórico. `fato_estoque_saldo` é
  "saldo hoje"; não existe "saldo de ontem" armazenado. Comparar saldo com
  período anterior é impossível com o cache atual e semanticamente sem sentido.
- `fato_estoque_movimento` é construído pelo worker **já agregado por mês**.
  Um toggle Dia/Semana exigiria reconstruir o fato em grão de dia — mudança de
  worker, fora do escopo de uma sub-fase de frontend.

Resultado: F3.5b entrega só o que o dado honestamente suporta — um seletor de
período **simples**, em grão de mês, sem tocar no worker nem nos fatos.

## Objetivo

Substituir o filtro de período atual (`PeriodFilter`, grão de mês, cru) por uma
barra de período moderna — pílulas de presets + calendário personalizado — nos
relatórios que têm dimensão temporal, com todo o estado na URL.

**Dois relatórios têm dimensão temporal** e recebem a barra:

1. **`entradas-saidas`** — "Entradas vs. saídas por mês" (LineChart sobre
   `fato_estoque_movimento`).
2. **`top-movimentados`** — "Top produtos movimentados" (BarChart sobre
   `fato_estoque_movimento`).

Ambos hoje declaram um filtro `{ tipo: "periodo", default: "3" }` e suas
queries (`getRelatorioEntradasSaidas`, `getRelatorioTopMovimentados`) já
filtram por `mes: { gte: periodoDe, lte: periodoAte }` — verificado em
`src/lib/actions/report-data.ts`.

Relatórios de snapshot (saldo, valor, produtos parados, concentração) **não**
têm dimensão temporal e **não** exibem a barra.

## Não-objetivos

- Comparação de períodos (removido — sem fundação de dados).
- Toggle de granularidade Dia/Semana/Mês (removido — exigiria mudança de fato).
- `KPICard` com delta ↑↓ (não nasce — dependia da comparação).
- Qualquer mudança no worker, nos builders de fato ou no schema Prisma.
- Granularidade abaixo de mês.

## Modelo de dados de período

Grão de **mês**. Um período é um par `{ de, ate }` de meses no formato
`YYYY-MM`, inclusivo nas duas pontas.

### Presets

| Preset (`periodo`) | Rótulo | Resolução |
|---|---|---|
| `mes` | Este mês | `de = ate = mês corrente` |
| `3meses` | Últimos 3 meses | `ate = mês corrente`, `de = mês corrente − 2` (3 meses inclusivos) |
| `ano` | Este ano | `de = janeiro do ano corrente`, `ate = mês corrente` |
| `tudo` | Tudo | `de`/`ate` = `null` → query não filtra por período |
| `custom` | Personalizado | `de`/`ate` vêm dos parâmetros `de` e `ate` da URL |

Preset padrão quando a URL não traz `periodo`: declarado no catálogo
(`temporal.periodoPadrao`). Para ambos os relatórios temporais: `3meses`.

> **Nota de comportamento.** O filtro atual (`default: "3"`) resolve
> `periodoDe = mesAtras(3)` … `periodoAte = mesAtual()` — uma janela de **4
> meses** (−3, −2, −1, 0). O preset `3meses` desta spec corrige para **3 meses
> inclusivos** (corrente, −1, −2), que é a leitura correta de "Últimos 3
> meses". A mudança de recorte é **intencional**, não uma regressão.

## Estado na URL

A `searchParams` é a **única fonte de verdade**. Deep-link e botão "voltar"
funcionam sem estado client-side paralelo.

| Param | Valores | Quando presente |
|---|---|---|
| `periodo` | `mes` \| `3meses` \| `ano` \| `tudo` \| `custom` | sempre que a barra escreve a URL |
| `de` | `YYYY-MM` | apenas quando `periodo=custom` |
| `ate` | `YYYY-MM` | apenas quando `periodo=custom` |

> Os parâmetros chamam-se `de`/`ate` (não os antigos `periodoDe`/`periodoAte`).
> A migração de nome faz parte desta sub-fase.
>
> `periodoParaParams` **sempre** emite `periodo` (e `de`/`ate` quando o preset
> é `custom`). Não há omissão "quando = padrão": a função é pura e não conhece
> o padrão do relatório; emitir o preset explícito é inofensivo e melhora a
> clareza do link compartilhado.

Regras de `resolverPeriodo`:
- `periodo` ausente → usa `temporal.periodoPadrao` do catálogo.
- `periodo=custom` → exige **`de` e `ate` ambos** presentes e válidos
  (`YYYY-MM`); se qualquer um faltar ou for inválido → cai para o preset
  padrão do catálogo.
- `de > ate` (em `custom`) → as duas pontas são trocadas (normalização
  silenciosa).
- Valor de `periodo` desconhecido → preset padrão.
- Mês posterior ao mês corrente em `de`/`ate` → recortado ao mês corrente.

## Camada de período — `src/lib/reports/periodo.ts`

Módulo de lógica **pura** (sem React, sem Prisma), 100% testável isolado.
É a **única fonte** de "mês corrente" e de aritmética de mês do sistema de
relatórios.

```ts
/** Presets escolhíveis na barra. */
export type PeriodoPreset = "mes" | "3meses" | "ano" | "tudo" | "custom";

/** Subconjunto válido como padrão de catálogo — "custom" exige de/ate. */
export type PeriodoPresetPadrao = Exclude<PeriodoPreset, "custom">;

/** Par de meses inclusivo; de/ate null = sem limite (preset "tudo"). */
export interface PeriodoResolvido {
  preset: PeriodoPreset;
  de: string | null;   // "YYYY-MM" ou null
  ate: string | null;  // "YYYY-MM" ou null
}

/** Mês corrente no formato "YYYY-MM", em UTC. `hoje` opcional para testes. */
export function mesCorrente(hoje?: Date): string;

/** Valida o formato YYYY-MM. */
export function ehMesValido(s: string): boolean;

/**
 * Resolve searchParams + padrão do catálogo num período concreto.
 * Aplica todas as "Regras de resolverPeriodo" da seção anterior.
 */
export function resolverPeriodo(
  params: { periodo?: string; de?: string; ate?: string },
  padrao: PeriodoPresetPadrao,
  hoje?: Date,
): PeriodoResolvido;

/**
 * Serializa um período num objeto de searchParams. Sempre inclui `periodo`;
 * inclui `de`/`ate` apenas quando `preset === "custom"`.
 */
export function periodoParaParams(p: PeriodoResolvido): Record<string, string>;

/**
 * Rótulo legível de um período resolvido:
 * - `tudo` (de/ate null) → "Tudo";
 * - mês único (de === ate) → "mar/2026";
 * - intervalo → "jan/2026 – mar/2026".
 * Consumido apenas pela pílula "Personalizado" da PeriodBar.
 */
export function rotuloPeriodo(p: PeriodoResolvido): string;
```

`resolverPeriodo` e `mesCorrente` aceitam um `hoje` opcional para os testes
fixarem a data. Todo cálculo de mês é feito em **UTC** (mesma convenção de
`filters.ts`).

## Catálogo

`ReportEntry` (`src/lib/reports/types.ts`) ganha um campo opcional:

```ts
/** Declara que o relatório tem dimensão temporal. Ausente = snapshot. */
temporal?: {
  periodoPadrao: PeriodoPresetPadrao;
};
```

- `entradas-saidas` e `top-movimentados` passam a declarar
  `temporal: { periodoPadrao: "3meses" }`.
- O filtro `{ tipo: "periodo", default: "3" }` é **removido** das
  `secoes[].filtros` dos dois relatórios — o período deixa de ser filtro de
  seção e vira dimensão do relatório. Resultado: `entradas-saidas` fica com
  `filtros: [{ tipo: "armazem" }]`; `top-movimentados` com
  `filtros: [{ tipo: "sentido" }]`.
- `ReportFilterTipo` perde o membro `"periodo"`.

## Componentes

Todos seguem o `ui-ux-pro-max` (tokens semânticos, foco visível, alvo de
toque ≥ 44px, `cursor-pointer`, `prefers-reduced-motion`). A skill
`ui-ux-pro-max` é consultada na task de cada componente.

### `PeriodBar` — `src/components/reports/period-bar.tsx`

- Client component. Renderiza a fileira de pílulas (uma por preset) + a pílula
  "Personalizado".
- Rótulo "PERÍODO" em `text-xs uppercase tracking-wide text-muted-foreground`.
- **Acessibilidade:** o conjunto é um seletor de escolha única —
  `role="radiogroup"` com `aria-label="Período"`; cada pílula é
  `role="radio"` com `aria-checked` refletindo o preset ativo. Navegação por
  teclado entre as pílulas (setas) conforme o padrão radiogroup.
- Pílula ativa: fundo violet (`bg-primary text-primary-foreground`); inativas:
  `bg-muted hover:bg-muted/70`. Transição 150–200ms.
- "Personalizado": pílula com ícone `CalendarDays` (lucide) que abre o
  `CustomRangePopover`. Quando o período é `custom`, a pílula exibe o intervalo
  via `rotuloPeriodo` (ex.: "jan/2026 – mar/2026").
- Ao escolher um preset/intervalo, faz `router.push` atualizando os params
  via `periodoParaParams`, **preservando os demais searchParams** (busca,
  `armazemId`, `sentido` etc.) — mesmo padrão de atualização de URL do
  `ReportFilters` (`setParam`: clona `searchParams`, sobrescreve só as chaves
  do período).

### `CustomRangePopover` — `src/components/reports/custom-range-popover.tsx`

- Popover do design system. base-ui **não** tem componente de calendário —
  esta sub-fase cria um wrapper mínimo `src/components/ui/popover.tsx` sobre
  `@base-ui/react/popover` (mesmo padrão de `select.tsx`/`dialog.tsx`), e o
  seletor de meses em si é um componente sob medida.
- Seleção de **mês inicial** e **mês final** via grade de 12 meses, com
  navegação de ano por chevrons. Não é calendário de dias — o grão é mês.
- O "mês corrente" usado para desabilitar meses futuros vem de
  `mesCorrente()` de `periodo.ts` — **nunca** de um `new Date()` local — para
  não divergir de `resolverPeriodo`.
- Botões `Cancelar` e `Aplicar`; "Aplicar" só habilita com intervalo válido
  (`de` e `ate` preenchidos, ambos ≤ mês corrente).
- Meses futuros ficam desabilitados. Nesta v1 não há data-limite mínima
  declarada no catálogo — qualquer mês ≤ corrente é selecionável.
- Fecha no `Esc` e no clique fora; foco volta ao gatilho.

### Integração — `report-view.tsx` / `page.tsx`

`page.tsx` (server component, `src/app/(protected)/relatorios/[id]/page.tsx`):

- Após obter `report`, se `report.temporal` existir, resolve o período **uma
  vez**: `const periodo = resolverPeriodo(sp, report.temporal.periodoPadrao)`.
  Caso contrário `periodo = null`.
- No laço sobre `report.secoes`, após `parseFilters(secao, sp)`, injeta o
  período antes de chamar a query:
  ```ts
  const filtros = parseFilters(secao, sp);
  if (periodo) {
    filtros.periodoDe = periodo.de ?? undefined;
    filtros.periodoAte = periodo.ate ?? undefined;
  }
  const resultado = await query(filtros);
  ```
- Passa `periodo` para `<ReportView … periodo={periodo} />`.

`ReportView` (`report-view.tsx`):

- `ReportViewProps` ganha o campo `periodo: PeriodoResolvido | null`.
- Renderiza `<PeriodBar periodo={periodo} />` no topo (acima do
  `ReportFilters`) **apenas** quando `periodo` não é `null`.
- A `PeriodBar` substitui o antigo `PeriodFilter`; os demais filtros continuam
  no `ReportFilters`.

## Queries

`ReportFilterValues` (`types.ts`): os campos `periodoDe`/`periodoAte`
continuam existindo e no formato `YYYY-MM`, mas passam a ser alimentados pela
injeção em `page.tsx` (acima) — não mais pelo parser de filtro de seção.

`getRelatorioEntradasSaidas` e `getRelatorioTopMovimentados`
(`src/lib/actions/report-data.ts`): **nenhuma mudança de assinatura nem de
corpo** — ambas já filtram `mes: { gte: periodoDe, lte: periodoAte }` apenas
quando os dois estão presentes. Com o preset `tudo` (ambos `null` →
`undefined` no `ReportFilterValues`), o recorte de período é omitido —
comportamento já existente.

`parseFilters` (`src/lib/reports/filters.ts`): o caso `"periodo"` é
**removido**. Os helpers `mesAtual`/`mesAtras` e `MES_REGEX` ficam órfãos após
a remoção (a lógica equivalente vive em `periodo.ts`) — devem ser removidos no
mesmo passo, confirmando por busca que nada mais os usa.

## Estados e erros

- **Sem dados no período**: o `ChartEmpty` (F3.5a) cobre — o gráfico mostra o
  empty state com a dica de ajustar o período.
- **Período inválido na URL**: normalizado silenciosamente por `resolverPeriodo`
  (ver "Regras de resolverPeriodo") — nunca quebra a página.
- **Preset `tudo`**: a query roda sem filtro de período; pode trazer todos os
  meses — aceitável (`fato_estoque_movimento` ≈ 12k linhas, poucas dezenas de
  meses agregados).

## Histórico de revisões

### Review #1 (SPEC v1 → v2) — auditoria adversarial

Achados materiais corrigidos (verificados contra o código real):

- **F1 — erro factual de escopo.** A v1 afirmava que "apenas `entradas-saidas`
  tem dimensão temporal". A leitura de `catalog.ts` revelou que
  **`top-movimentados` também declara `{ tipo: "periodo" }`** e sua query
  filtra por `mes`. A v2 inclui os dois relatórios em todo o documento.
- **F2 — tipo frágil.** `temporal.periodoPadrao` aceitava `PeriodoPreset`
  inteiro, incluindo `custom` — inválido como padrão. Criado
  `PeriodoPresetPadrao = Exclude<PeriodoPreset, "custom">`.
- **F3 — integração vaga.** A v1 dizia que `page.tsx` "injeta o período" sem
  dizer como.
- **F4 — off-by-one não documentado.** O filtro atual (`default: "3"`) produz
  janela de 4 meses; o preset `3meses` produz 3.
- **F5 — premissa de UI.** base-ui não tem componente de calendário.

### Review #2 (SPEC v2 → v3) — auditoria crítica profunda

> A primeira tentativa de review #2 foi rasa (achados só de superfície) e foi
> **descartada**. Esta é a revisão profunda que a substituiu.

- **R2-A — `custom` parcial indefinido.** A regra "sem `de`/`ate` válidos →
  padrão" não cobria o caso de só um dos dois ser válido — comportamento
  indefinido. A v3 fixa: `custom` exige os **dois** válidos; qualquer falha →
  preset padrão.
- **R2-B — fonte dupla de "mês corrente".** `periodo.ts` calcula em UTC; o
  `CustomRangePopover` (client) usaria `new Date()` local — perto da virada do
  mês, popover e `resolverPeriodo` discordariam. A v3 cria `mesCorrente()` em
  `periodo.ts` como única fonte e manda o popover consumi-la.
- **R2-C — `rotuloPeriodo` sem comportamento definido** para `de`/`ate` null
  (`tudo`) e mês único. A v3 define os três casos e documenta que só a pílula
  "Personalizado" o consome.
- **R2-D — integração não fechada.** A interface `ReportViewProps` não
  declarava o novo campo `periodo`, e a injeção do período nos
  `ReportFilterValues` não estava expressa em código. A v3 mostra o trecho
  exato de `page.tsx` e a mudança de props.
- **R2-F — acessibilidade incorreta.** `aria-pressed` em 5 pílulas de seleção
  única faz o leitor de tela anunciar 5 toggles independentes. A v3 adota o
  padrão correto `role="radiogroup"`/`role="radio"`/`aria-checked`.
- **R2-G — inconsistência interna.** A tabela de URL dizia "`periodo` presente
  sempre que ≠ padrão", mas `periodoParaParams` é função pura e não conhece o
  padrão do relatório — não tem como decidir omitir. A v3 resolve:
  `periodoParaParams` sempre emite `periodo`.
- Achado adicional do popover: foi explicitado que a sub-fase cria também um
  wrapper mínimo `ui/popover.tsx` (não existia no design system).

Após R2-A…G a spec foi relida inteira: escopo enxuto, sem exagero, sem
conceito quebrado, sem requisito ambíguo restante. **Spec liberada para o
plano.**
