## §1. Problema

A plataforma tem **duas camadas de faturamento que não conversam**, e a tool mais usada
("qual o faturamento?") responde **inflada**:

| Camada | Onde | Base | Filtro de venda | Intercompany |
|---|---|---|---|---|
| **Antiga** | `src/lib/reports/queries/fiscal.ts` | `nota.vrNf` | por **natureza** (`idsNaoVenda`) | **NÃO elimina** |
| **Canônica** | `src/lib/metrics/fiscal/` (Fases 1-2) | `item.vrProdutos` | por **CFOP** (`ehReceita`) | **ELIMINA** |

**Impacto medido (2025, reproduzido no cache real pela review):**
`fiscal_faturamento_periodo` (antiga) = **R$ 551.251.936,84** (5.017 notas); receita
externa real (canônica) = **R$ 325.481.428,48**. **Divergência R$ 225.770.508 (+69,4%)**.
Decomposição: **R$ 217,9 mi (96,5%) é intercompany não eliminado** + **R$ 7,9 mi é a troca
do filtro de operação (natureza→CFOP)**. A base vrNf→vrProdutos isolada vale só **-0,27%**.
O dono quer o número real.

**Achado R-intercompany-fallback-fragil:** parte da eliminação intragrupo (em 2025, **783
notas** marcadas **só** pelo regex de nome , acumulado 13 anos = 2.538 notas / R$ 278,8 mi)
depende do CNPJ embutido em `participante_nome`, porque `fato_parceiro` está corrompido para
os estabelecimentos do grupo (`documento_digits` vazio , confirmado nos pid 2, 9, 10, 11,
12, 13). O número de hoje está correto, mas frágil.

**Achado R-periodo-acumulado:** toda tool fiscal sem período soma o cache inteiro (~13 anos).
As 3 tools da jornada já usam `resolverPeriodoFiscal`; as demais ainda não.

### Premissas corrigidas (vs enunciado / vs v1)

1. A camada antiga **não** tem "default 13 anos": `buildPeriodoWhere` retorna `{}` (todo o
   histórico) quando o par período está incompleto. Efeito igual, correção = `resolverPeriodoFiscal`.
2. O **dashboard (Next.js) não importa** `reports/queries/fiscal.ts`. Todo consumo passa pelo
   **MCP**; corrigir as tools propaga para Nex/WhatsApp/Playground.
3. A base `item.vrProdutos == nota.vrProdutos (cabeçalho)` ao centavo, **0 itens sem nota e
   0 notas sem item** no recorte de venda autorizada , refuta a preocupação de distorção por órfãos.

## §2. Decisões de produto (confirmadas pelo usuário + refinadas pela review fiscal)

1. **Headline do grupo = receita externa real (sem intercompany).** Sem filtro de empresa, o
   número principal é a **receita externa consolidada (CPC 36)**; o faturamento bruto/individual
   (com intragrupo) é **linha de auditoria secundária**, sempre rotulado.
2. **Por empresa específica = NÃO eliminar, MAS exibir com honestidade.** O usuário escolheu
   "receita real da CNPJ" (não eliminar) , mantido. **Refinamento obrigatório (review fiscal A1):**
   como há estabelecimentos-concentradores (a Jds Matriz é 94,8% intragrupo), o envelope por
   empresa retorna **`individual`, `externa` e `intragrupoEliminavel` com PARIDADE de destaque**
   (não um único número grande). Quando `percentualEliminado > 0,5`, a resposta rotula a entidade
   como **"estabelecimento concentrador (maioria das vendas é intragrupo)"**. Assim preserva-se a
   receita real da CNPJ sem recriar a inflação que a fase mata.
3. **Escopo = núcleo P1 + sentinelas.** Itens 1 (unificar tools de faturamento), 2 (whitelist
   intercompany), 3 (fix período) + sentinelas de conferência (S0/S1/S2/S3 + guarda autorizada).
   Transparência sem-CFOP/5949 por finalidade (item 5) entra **se couber**; as demais 9 checagens
   do item 4 vão para a **Fase 2.6**.

## §3. Objetivo e não-objetivos

**Objetivo:** que qualquer pergunta de faturamento (Nex in-app, WhatsApp/n8n, Playground)
responda o número real (externa como headline do grupo; individual honesto por empresa), com
período sempre explícito, marcação intercompany resiliente (whitelist → cadastro → nome), tudo
travado por conferência (TS vs SQL bruto) e E2E contra o cache real.

**Não-objetivos (YAGNI):**
- Não criar nem remover tools (mantém **104** no `integration.test.ts`).
- Não alterar `id`, `descricao`, `triggers` nem `inputSchema` das tools migradas (preserva
  roteamento, snapshot de catálogo e frozen-30). A mensagem nova vai no `aviso`/formatador.
- Não tocar tools que não são faturamento de venda além do fix de período elegível.
- Sem migration de schema (duas queries nativas + join em memória). Sem contábil/DRE/lucro.
- Devolução como dedução de receita continua na Fase 3.

## §4. Arquitetura da solução

### §4.1 Princípio: uma definição canônica de "faturamento de venda"

Fonte única = **`item.vrProdutos` classificado por CFOP** (`ehReceita ∈ {venda, exportacao,
servico}`), com eliminação intragrupo disponível. Invariante travado (conferência I3):
`receitaIndividualTotal` (F2) == `F1.totalReceita`. Trocar a base de `vrNf` para `vrProdutos`
nas tools de faturamento muda o número individual em **-0,27%** (não impostos , é desconto de
cabeçalho); a diferença maior (~R$ 7,9 mi) vem da troca **natureza→CFOP**. Tudo documentado em
RADAR e travado por baseline.

### §4.2 Core compartilhado (nova unidade, reusável)

`src/lib/metrics/fiscal/_itens-venda-grupo.ts`:

```ts
carregarItensVendaComGrupo(prisma, input: FaturamentoInput): Promise<{
  itens: ItemVendaGrupo[];          // por grupo (documentoId, cfopId): valorProdutos, ehReceita,
                                    // intragrupo, participanteId, participanteNome, empresaId,
                                    // empresaNome, mesEmissao
  marcacaoPorNota: Map<number, {    // CHAVE = odooId da nota (preserva a contagem atual)
    intragrupo: boolean; participanteId: number|null; participanteNome: string|null;
    empresaId: number|null;
  }>;
  participantesGrupo: Set<number>;
}>
```

Requisitos do contrato (review arq A4):
- Mantém **exatamente** o padrão atual de `receitaConsolidada`: `groupBy(["documentoId","cfopId"])`
  no item + `findMany` de notas (`fato_nota_fiscal`), marcação por `odooId`, join em memória por
  `documentoId == odooId`, `Number(Decimal)`. **Sem `$queryRaw`.**
- A **contagem de notas** (`notasIntragrupo`/`notasExternas`) é derivada de `marcacaoPorNota`
  (notas distintas), **não** reconstruída a partir de `itens` , garante paridade com hoje
  (notas sem item continuam contadas).
- `receitaConsolidada`, `faturamentoSerieMensal` e `faturamentoPorClienteCanon` consomem o MESMO
  core → período, base, situação e definição de intragrupo idênticos.

### §4.3 Métricas canônicas (refatorar 1, criar 2)

- **`receitaConsolidada`** (existe): refatorar para consumir o core. **Saída byte-idêntica**
  (travada pelo teste unitário existente + conferência I3/I4 nos **5 períodos**, não só 2025).
  O teste unitário hoje mocka `fatoParceiro.findMany`; o refactor pode reordenar queries , prever
  ajuste de mock como parte do refactor (não é regressão).
- **`faturamentoSerieMensal`** (novo): por mês `{ mes, individual, externa, intragrupoEliminavel,
  notasExternas }` + agregados `{ totalIndividualAno, totalExternaAno, totalNotasExternasAno }`
  (os agregados que o formatador precisa , ver §4.8).
- **`faturamentoPorClienteCanon`** (novo): ranking por participante; marca cada cliente
  `grupo|externo`; retorna `linhas` (externos, desc, paginado) + `totalExterno` + `totalIntragrupo`
  + `topClienteExterno`.

### §4.5 Blindagem da marcação intercompany (item 2)

`src/lib/fiscal/grupo/whitelist-grupo.ts`:

- **`PARTICIPANTES_GRUPO_WHITELIST: ReadonlySet<number>` = `{2,9,10,11,12,13,14,15,16,19,20,21,22,23}`**
  , validados no cache real pela review (aparecem como participante em notas intragrupo, nome com raiz
  do grupo). **Excluídos explicitamente (comentário no arquivo):** 8722 (Jaguaribe), 8723 (Vilmar),
  9552 (Smartfit), 7719 (Residencial) , odoo_id **reciclados**, 0 notas, NÃO são o grupo. Matrix Fit
  franquias (32493616/50075046/57692916) são clientes externos, **não** entram.
- **Critério de inclusão (reproduzível):** `participante_id` que aparece em ≥1 nota cujo
  `participante_nome` casa `RAIZES_GRUPO` por CNPJ, **e** cujo `documento_digits` (quando presente)
  tem raiz ∈ `RAIZES_GRUPO`. Curadoria deve ser re-derivável por SELECT, não lista mágica.
- **`ehNotaIntragrupo` evoluído** , ordem: (1) `participanteId ∈ whitelist`; (2) `participanteId ∈
  participantesGrupo` (dinâmico, doc do cadastro); (3) raiz(`participanteNome`) ∈ `RAIZES_GRUPO`
  (fallback, última defesa). **Nunca** `fato_parceiro.eh_empresa`.
- **Invariante (corrigido pela review):** delta esperado **= R$ 0 hoje** (o fallback de nome já
  captura tudo) , a whitelist é **resiliência**, não correção. A eliminação **não pode diminuir**
  (travado por S0, §4.6).

### §4.6 Sentinelas de conferência (P1) , primitivas e checagens

Expandir `scripts/conferencia-fiscal.ts`. Adicionar primitiva de **banda** (review arq A1), separada
do `check` de igualdade (`TOL=0.5`):

```ts
checkBanda(nome, valor, min, max, erros)   // contagem/valor dentro de [min,max]; fora → erro
```

Checagens (por período, como a conferência já itera 2023/24/25/26/acumulado):
- **S0 , eliminação não reduz (gate):** grava `receitaIntragrupoEliminavel` por período **antes** da
  whitelist num JSON dedicado (`docs/superpowers/research/baseline-eliminacao-pre-whitelist.json`);
  a conferência **falha** se o valor pós-whitelist for **menor** que o baseline. É a única prova de
  que a whitelist não reduziu.
- **S1 , fallback só-por-nome (banda):** contar notas marcadas intragrupo **só** pelo regex de nome
  (não whitelist nem cadastro). Banda **por período** (2025 ≈ 783 notas; acumulado ≈ 2.538). Alerta
  (não gate) se saltar acima da banda , sinaliza regressão de cadastro no Odoo.
- **S2 , divergência nome×cadastro (banda):** participantes cujo nome tem raiz do grupo mas
  `fato_parceiro` aponta doc de fora (hoje ~10 pares). Alerta se crescer.
- **S3 , base individual fecha (gate):** por ano, `individual == externa + intragrupoEliminavel`
  (reforça I4 com a whitelist no caminho).
- **S4 , guarda situação=autorizada (gate, promovido a P1):** nenhuma métrica de faturamento de venda
  pode somar `em_digitacao` (2025 = R$ 14,5 mi). Confronta `receitaIndividualTotal` contra SQL bruto
  com `situacao_nfe='autorizada'` , se um filtro vazar, diverge.

Decisão gate vs alerta: S0/S3/S4 são **gate** (exit 1); S1/S2 são **alerta** (log, não falham).

### §4.7 Transparência receita-perdida (item 5 , leve, se couber)

Decompor por `finalidade_nfe`, em runtime (números ilustrativos de 2025, não fixos):
- `sem_cfop`: ~R$ 5,8 mi finalidade=1 (venda candidata) + ~R$ 0,8 mi finalidade=4 (devolução).
- **`outras` CFOP 5949/6949** (review fiscal A4): ~R$ 6,3 mi finalidade=1 (saída normal) , possível
  venda real fora do headline. Exibir as duas fontes por finalidade; manter fora da receita
  (conservador) mas visível. Total receita-perdida candidata 2025 ≈ R$ 12 mi.

### §4.8 Contrato de `_DESTAQUE` e headline condicional (review arq B2/B3)

**Quem decide o headline: o handler** (não o formatador). Mecanismo:
- `montarEscopoEmpresa` retorna `escopo.empresaId` = `undefined` (grupo) ou número (empresa).
- O handler popula `_DESTAQUE` com chaves **estáveis**:
  - `headlineValor` (number), `headlineRotulo` (string): grupo → externa / "receita externa real";
    empresa → individual / "faturamento da empresa (inclui intragrupo)".
  - secundários sempre presentes: `receitaExterna`, `receitaIndividual`, `intragrupoEliminavel`,
    `percentualEliminado`, `notasExternas`, `concentrador` (bool, `percentualEliminado>0,5`),
    `periodoLabel`.
- O **formatador é burro**: lê `headlineValor/headlineRotulo` + secundários, formata headline + linha
  de auditoria; quando `concentrador`, imprime o aviso de "estabelecimento concentrador".

Reescrever os 3 formatadores para o novo contrato (eles hoje leem `valorFaturado`/`totalNotas`/
`topCliente`/`totalAno` que **deixam de existir** , senão `formatBRL(0)` mudo). Travar cada um com
teste de unidade (input `_DESTAQUE` novo → string esperada), incluindo os dois caminhos de headline.

## §5. Fluxo de dados (tool de faturamento, pós-unificação)

```
Nex / WhatsApp / Playground
  → fiscal_faturamento_periodo(periodoDe?, periodoAte?, empresaRef?)
     → montarEscopoEmpresa(prisma, empresaRef)          // escopo.empresaId: undefined|number
     → resolverPeriodoFiscal(de, ate)                    // default ano corrente + label
     → receitaConsolidada(prisma, { periodoDe, periodoAte, empresaId })
        → carregarItensVendaComGrupo(...)                // core: groupBy itens + findMany notas
           → ehNotaIntragrupo(nota, participantesGrupo)  // whitelist → cadastro → nome
     → handler decide headline (empresaId? individual : externa) → _DESTAQUE (§4.8)
     → withFreshness([...]) + enriquecerEnvelope(...)
  → fmtFaturamentoPeriodo(env) → headline + auditoria + período + (aviso concentrador)
```
