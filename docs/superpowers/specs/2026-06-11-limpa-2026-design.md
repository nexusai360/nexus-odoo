# SPEC v3 , Limpa 2026+ (cache só com dados de 2026 em diante)

> Pedido do usuário (2026-06-11, prioridade máxima): cache apenas com dados de
> 2026 em diante, "limpa completa, tabela a tabela", liberar espaço.
> **v3 = v1 + 2 reviews adversariais aplicadas (4 BLOCKERs + 8 MAJORs).**
> Reviews completas nos transcripts; achados decisivos incorporados abaixo.

## 0. Os 4 BLOCKERs que derrubaram a v1 (e como a v3 resolve)

1. **Corte de título por data deletaria R$ 118,1mi de DÍVIDA VIVA** (medido:
   1.829 títulos abertos/provisórios com vencimento pré-2026, sendo a_pagar
   R$ 118.108.154,18 , a espinha do "a pagar" da Johnson). → **Título corta por
   SITUAÇÃO, nunca por data pura** (§2.2).
2. **Incremental filtra por `write_date`** , registro de 2024 editado em 2026
   re-entraria. → Filtro de data de negócio vira **cláusula AND adicional**
   permanente no domain (§3.1).
3. **Snapshot faz full-refresh com domínio vazio** (reimporta 100% do histórico
   a cada 30min) + guarda CR-02 aborta com 0 linhas. → Filtro no
   `searchReadPaged` + classificação modelo a modelo + CR-02 ciente do corte (§3.2).
4. **`rawDeleted=true` NÃO libera espaço** (UPDATE aumenta bloat; usuário quer
   espaço). → **DELETE físico + `VACUUM (FULL, ANALYZE)`** com medição
   `pg_total_relation_size` antes/depois (§4).

## 1. Decisões de design (v3)

1. **DELETAR fisicamente, não arquivar.** O Odoo é a fonte permanente; nada se
   perde. `pg_dump` das tabelas tocadas ANTES do purge = rede de segurança
   barata (rollback sem resync de horas).
2. **Critério por CLASSE de tabela** (inventário §2), nunca um critério único.
3. **Datas-fonte com TRÊS nomes por modelo** (achado da review): campo no ODOO
   (para o domain do sync), chave no JSON do raw (`data->>'x'`) e coluna do
   fato. O `MODEL_CATALOG` ganha `corte?: { odoo: string; raw: string;
   fato?: string }` , fonte única lida pelo worker E pelo purge.
4. **NULL nunca deleta**: linha transacional com campo de corte NULL permanece
   e é LOGADA no relatório para inspeção (nunca deletar no escuro).
5. **Plataforma fora do escopo** (nomeado): users, conversas, perícias,
   `llm_usage`, `mcp_audit_log`, feedbacks, router decisions , não são ERP.

## 3. Filtro permanente no SYNC (senão reimporta)

1. **Incremental**: `domain = [[corte.odoo, ">=", "2026-01-01"],
   ...(since ? [["write_date", ">", ...]] : [])]` , a data de negócio é
   cláusula ADICIONAL permanente, em todo ciclo e no backfill. Para
   `finan.lancamento`, o domínio NÃO usa data pura (espelha §2.2: estado
   quitado/baixado + pago antes de 2026 fica fora; vivos entram sempre).
2. **Snapshot**: mesma cláusula no `searchReadPaged`; modelos foto-atual ficam
   sem corte; `raw_estoque_extrato` reclassificado. **Guarda CR-02**: 0 linhas
   legítimo pós-corte NÃO é glitch (a guarda compara com o cache JÁ purgado).
3. **Reconcile**: `searchIds` com a MESMA cláusula , o conjunto "vivo" fica
   coerente com o cache filtrado; `rawDeleted` volta a significar apenas
   "deletado de verdade no Odoo (2026+)". A premissa da v1 sobre o reconcile
   estava errada (review #1.3) e foi removida.
4. **Validar campo filtrável no Odoo** modelo a modelo antes de cravar (alguns
   são computados/views).

## 4. Execução em PROD (ordem obrigatória)

1. Deploy do filtro de sync (catálogo + worker) → observar 1 ciclo: ZERO linha
   nova pré-2026.
2. Pausar o worker (fila BullMQ) na janela do purge.
3. `pg_dump` das tabelas a purgar (rede de segurança).
4. `scripts/purge-pre-2026.ts --apply` (dry-run é o default; `--apply` SÓ com
   aprovação humana explícita do relatório do dry-run): DELETE físico, filhos →
   pai, NULL preservado e logado.
5. `VACUUM (FULL, ANALYZE)` por tabela tocada (trava tabela; janela calculada ,
   923MB+213MB estimados) com `pg_total_relation_size` antes/depois no relatório.
6. Rebuild de TODOS os fatos; invariante financeiro (§2.2) conferido.
7. Religar worker → validar 2 ciclos sem reimport.
**Rollback**: restaurar do pg_dump (minutos); último recurso = resync completo
do Odoo (horas, fonte intacta).
