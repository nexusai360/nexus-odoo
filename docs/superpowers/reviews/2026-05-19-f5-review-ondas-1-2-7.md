## ALTO

### ALTO-1 — Gemini/OpenRouter quebram com o JSON Schema cru do MCP
**Arquivo:** `src/lib/agent/llm/providers/gemini.ts:99-110` (`mapTools`) e
`src/lib/agent/mcp-client.ts:34-40` (`mcpToolsToProviderTools`).

`mcpToolsToProviderTools` repassa o `inputSchema` do MCP **sem nenhuma limpeza**
como `parameters`. O MCP da F4 registra as tools via `registerTool` do
`@modelcontextprotocol/sdk` a partir de shapes Zod — o `tools/list` resultante
devolve JSON Schema padrão contendo `$schema`, `additionalProperties` e
potencialmente `$ref`/`format`. O `mapTools` do Gemini joga esse objeto direto em
`functionDeclarations[].parameters`. A API do Gemini aceita **apenas** o
subconjunto OpenAPI 3.0 — `$schema` e `additionalProperties` provocam **HTTP 400**.
OpenRouter, ao rotear para back-ends Google, sofre o mesmo. Anthropic e OpenAI
toleram campos extras, por isso o provider default não falha — e como a e2e rodou
só com MOCK, o bug está latente.

**Evidência:** `mcp/server.ts:143-148` registra com `inputSchemaShape` (Zod shape)
→ o SDK serializa JSON Schema completo. Nenhum teste exercita um provider real
contra esse schema.

**Recomendação:** sanitizar o schema em `mcpToolsToProviderTools` (ou num
sanitizador por provider): remover `$schema`, `additionalProperties`, `$ref`
resolvido inline, e mapear `format` não suportado. Adicionar teste com um
`inputSchema` realista (com `$schema`/`additionalProperties`) verificando que o
`mapTools` de cada provider produz schema aceito.

### ALTO-2 — Streaming do Anthropic é desligado sempre que há tools
**Arquivo:** `src/lib/agent/llm/providers/anthropic.ts:151`.

`const useStream = request.stream === true && !request.tools?.length;` — o
streaming token-a-token é desativado **sempre que `tools` está presente**. No
`runAgent` as tools do MCP estão sempre presentes (`tools` nunca é vazio quando o
MCP está no ar). Resultado: **o turno final do assistente nunca streama** no fluxo
real do agente, contrariando a SPEC §8.1 ("no turno final do assistente, streama o
texto" / "streaming token-a-token incluído no escopo para o provedor default").
O streaming só funcionaria num cenário sem tools, que não ocorre em produção.

**Recomendação:** o streaming da Anthropic suporta `tool_use` em SSE. Habilitar
stream mesmo com tools e parsear blocos `content_block_start`/`_delta` de
`tool_use` no `#parseStream` (hoje ele só trata `text_delta`). Alternativa
mínima: streamar apenas a última iteração (quando o modelo não pede mais tools) —
mas isso exige reestruturar o loop. Como onda 3 depende disso, registrar a
limitação explicitamente se não for corrigido agora.

### ALTO-3 — `calculateCost` ignora `perMinuteUsd` no caminho real → custo zero para transcrição
**Arquivo:** `src/lib/agent/llm/catalog.ts:253-264` + `transcribe.ts`.

`whisper-1` tem `pricing: { inputPerMTok: 0, outputPerMTok: 0, perMinuteUsd: 0.006 }`.
`calculateCost` só usa `perMinuteUsd` quando recebe `extras.durationMs`. Mas
`logUsage` (`usage-logger.ts:43`) **nunca passa `extras`** — chama
`calculateCost(model, tokensIn, tokensOut)` com 3 argumentos. Logo, para
`whisper-1`, `costKnown=true` e `costUsd = (0·0 + 0·0)/1e6 = 0`. Isso é
exatamente o "custo zero silencioso" que o BUG 2 deveria eliminar (SPEC §4.6.2):
o modelo tem preço, o custo real não é zero, mas grava 0 com `costKnown=true`.
A transcrição via `whisper-1` aparece como gratuita no consumo.

**Recomendação:** `logUsage` deveria aceitar e repassar `durationMs` ao
`calculateCost` para modelos `perMinuteUsd` (a interface `LogUsageArgs` já tem
`durationMs`, mas ele é gravado na row, não passado ao cálculo de custo). Ou
marcar modelos `perMinuteUsd` como não-token e calcular custo no `transcribe.ts`.

## MÉDIO

### MÉDIO-1 — `getActiveLlmConfig` não garante config única; corrida não tratada
**Arquivo:** `src/lib/agent/llm/get-active-config.ts:40-52`.

Usa `findFirst({ where: { isActive: true }, orderBy: { updatedAt: 'desc' } })`. A
SPEC §9.2 (B13) exige que ativar uma config seja transacional (desativa todas +
ativa a escolhida). Se a action de ativação (onda 3) não for atômica, podem
existir duas rows `isActive=true` e o `findFirst` escolhe silenciosamente uma. Não
há `@@unique` parcial nem constraint no schema garantindo unicidade de `isActive`.
A onda 1 não tem como garantir isso sozinha, mas o leitor deveria ao menos logar
um aviso se `count(isActive=true) > 1`.

**Recomendação:** quando a Task 3.0a for implementada, conferir a transação; aqui,
trocar `findFirst` por `findMany` + aviso quando houver mais de uma ativa.

### MÉDIO-2 — Trava de drift do BI schema só compara nomes de tabela
**Arquivo:** `src/lib/agent/bi-schema-reference.test.ts:37-55`.

A Task 1.9 / SPEC §4.5.2 (B6) pede que o teste compare a referência com as fact
tables reais — "nomes de tabela, colunas e tipos". O teste atual só verifica que
cada `@@map("fato_*")` aparece como substring na constante e o reverso. **Drift de
coluna passa despercebido**: se um campo `vr_saldo` for renomeado ou removido de um
`Fato*` no schema, `BI_SCHEMA_REFERENCE` fica desatualizado e o agente gera SQL
inválido no Caminho 3c — e o teste continua verde. A trava cumpre só metade do
contrato.

**Recomendação:** estender o teste para extrair os campos de cada modelo `Fato*`
do schema e verificar que cada coluna aparece na seção correspondente da
constante.

### MÉDIO-3 — Default de `kbEnabled` divergente entre schema e fallback do orquestrador
**Arquivo:** `src/lib/agent/run-agent.ts:107-118` (`loadAgentSettings`).

O fallback quando não existe row `AgentSettings` usa `kbEnabled: false` e
`suggestionsEnabled: false`. O schema Prisma (`schema.prisma:1628-1629`) declara
ambos `@default(true)`. Numa instância nova, antes de qualquer escrita em
`AgentSettings`, o agente roda com KB e sugestões **desligadas**, e assim que a
primeira gravação criar a row eles passam a **ligados** — comportamento muda sem
ação do usuário. Inconsistência de premissa.

**Recomendação:** alinhar o fallback ao default do schema (`true`/`true`), ou
criar a row singleton via seed/migration para nunca depender do fallback.

### MÉDIO-4 — `searchKb` no fallback ignora `kbEnabled` e injeta todos os docs
**Arquivo:** `src/lib/agent/rag/search.ts:101-123` + `run-agent.ts:157-166`.

Quando não há credencial de embedding, `searchKb` cai no fallback que retorna
**todos** os `KbDocument` (`take: topK`, `orderBy: createdAt desc`) truncados.
Isso é o "texto integral truncado" previsto (SPEC §4.8, Q4) — ok em si. Mas o
fallback **não filtra por relevância** e o `runAgent` só chama `searchKb` se
`kbEnabled` — então está coerente. O problema real: no fallback, `topK=5` limita a
5 docs mais recentes; se a KB tiver mais de 5 docs, os demais nunca entram no
prompt mesmo sem busca semântica. O Nex original injetava a KB inteira (dentro do
budget de 30KB). Aqui, sem embedding, KBs com >5 docs perdem conteúdo
silenciosamente.

**Recomendação:** no fallback, não aplicar `take: topK` — buscar todos e deixar o
budget de 30KB do `composeSystemPrompt` fazer o corte (que é o comportamento
esperado do "texto integral truncado").

### MÉDIO-5 — Sem índice vetorial em `kb_documents.embedding`
**Arquivo:** migration `20260519054910_f5_pgvector/migration.sql`.

A coluna `embedding vector(1536)` foi criada sem índice `ivfflat`/`hnsw`. O
`searchKb` faz `ORDER BY embedding <=> $1` — sem índice, é full scan sequencial a
cada pergunta. Para a KB pequena prevista nesta fase é tolerável, mas é uma dívida
silenciosa: quando a KB crescer, a latência degrada sem aviso. O plano (Task 7.1)
não pediu índice explicitamente, então não é divergência do plano — é um gap de
qualidade.

**Recomendação:** adicionar `CREATE INDEX ... USING hnsw (embedding vector_cosine_ops)`
numa migration, ou registrar como dívida consciente no RADAR.

### MÉDIO-6 — `transcribe.ts` diverge do contrato do plano (Task 1.15)
**Arquivo:** `src/lib/agent/transcribe.ts:52`.

O plano especifica `transcribe(audioBytes, mimeType)` que lança
`TranscriptionUnavailable` quando não há credencial OpenAI. O implementado é
`transcribeAudio(audio: Blob, language)` que lança `Error` genérico. Não há classe
`TranscriptionUnavailable`. Callers da onda 3/4 que tentarem `instanceof
TranscriptionUnavailable` (análogo ao `EmbeddingUnavailable` do RAG) não vão
compilar/funcionar. Divergência de assinatura e de tipo de erro.

**Recomendação:** alinhar — ou atualizar o plano, ou criar a classe de erro e a
assinatura previstas. Como o RAG seguiu o padrão `EmbeddingUnavailable`,
consistência manda criar `TranscriptionUnavailable`.

## BAIXO

### BAIXO-1 — `logAudit` sem `await` em `user-whatsapp.ts`
**Arquivo:** `src/lib/actions/user-whatsapp.ts:119,158`.

`addWhatsappNumber`/`removeWhatsappNumber` chamam `logAudit({...})` sem `await`
(fire-and-forget). Em `credentials.ts` o `logAudit` é aguardado. Numa Server
Action serverless, a promise pendente pode não completar antes do retorno → risco
de auditoria perdida. Inconsistente com o resto do código.

**Recomendação:** `await logAudit(...)`.

### BAIXO-2 — `deleteCredential` não verifica existência antes do delete
**Arquivo:** `src/lib/agent/llm/credentials.ts:139-151`.

Se o `id` não existir, `prisma.llmCredential.delete` lança `P2025` (erro Prisma
cru) em vez de uma mensagem de domínio. `removeWhatsappNumber` faz o `findUnique`
antes — inconsistente.

**Recomendação:** `findUnique` antes, ou capturar `P2025` e devolver erro claro.

### BAIXO-3 — `costUsd` somado como 0 quando `costKnown=false` mascara o desconhecido
**Arquivo:** `src/lib/agent/run-agent.ts:212` e `types.ts:55`.

`totalUsage.costUsd += result.usage.costUsd ?? 0` — o total de custo de uma run
soma 0 para iterações de modelo sem pricing. O `LlmUsage` individual grava
`costUsd=null`/`costKnown=false` corretamente (BUG 2 ok no logger), mas o
`ChatUsage` agregado devolvido por `runAgent` perde a distinção. Quem consumir
`result.usage.costUsd` vê um número que pode estar subestimado sem sinalização.
Menor porque a tela de consumo lê de `LlmUsage` (correto), não do retorno.

**Recomendação:** adicionar `costKnown`/`costPartial` ao `ChatUsage` agregado, ou
documentar que o número agregado é só indicativo.

### BAIXO-4 — `normalizeE164` aceita comprimentos fora do previsto para BR
**Arquivo:** `src/lib/whatsapp/resolve.ts:24-57`.

A faixa aceita é 10–15 dígitos após o `+`. Um número nacional sem `+` e sem `55`
com, p.ex., 10 dígitos vira `+55` + 10 = 12 dígitos — aceito como fixo BR. Mas a
função também aceita `+1234567890` (10 dígitos, EUA) sem validar plausibilidade
de DDI. Para a 2ª linha de defesa (SPEC §5.3) isso é benigno — número
desconhecido cai em `unknown` de qualquer forma — mas a normalização pode produzir
E.164 "válido" para lixo. Aceitável; anotado para consistência.

**Recomendação:** nenhuma ação obrigatória; a resolução por `findUnique` cobre.

## O que está correto e bem feito

- **Identidade/RBAC (decisão #10):** `mcp-client.ts` injeta `userId` via header
  `x-mcp-user-id` — bate exatamente com `mcp/server.ts:192`. O número de WhatsApp
  nunca entra no `runAgent`; `resolveWhatsappUser` devolve `User`, e só o `userId`
  segue. Contrato respeitado.
- **Sessão MCP por invocação (B1):** `createMcpSession` por chamada, `close()` em
  `finally` no `runAgent`. Sem pool global. Correto.
- **BUG 2 (custo zero silencioso):** `calculateCost` devolve `{costUsd:null,
  costKnown:false}` para `pricing:null`; o `usage-logger` grava `costUsd=null`.
  Correto para modelos token-based (ver ALTO-3 para a exceção `perMinuteUsd`).
- **BUG 3 (fonte única):** catálogo+pricing num único array `MODELS`; sem tabela
  de preços separada. Correto.
- **BUG 5/6 (cotação):** `exchange-rate.ts` nunca retorna null — cache stale ou
  fallback fixo, sempre com `stale`/`spread` no objeto. `RATE_SPREAD` versionado e
  gravado em `rateSpread`. Correto.
- **BUG 7 (`isPlayground`):** presente no schema desde a migration `f5_agent_core`.
- **Cifra reversível:** `LlmCredential.encryptedApiKey` via `encrypt`/`decrypt`
  AES-256; `last4` em claro; `getActiveLlmConfig` decifra só em memória,
  `getPublicActiveLlmConfig` nunca expõe a chave. Correto.
- **SQL do RAG:** `searchKb`/`ingestKbDocument` usam `$queryRaw`/`$executeRaw`
  tagged templates — o `vectorLiteral` entra como parâmetro, sem concatenação.
  Sem injeção. O operador `<=>` (cosseno) está correto.
- **Dimensão 1536:** `embed.ts` valida `vector.length === 1536` antes de retornar;
  coluna confirmada `vector(1536)` no banco real.
- **Guard de tamanho de tool (B8):** `guardToolResult` trunca a 24KB com aviso,
  byte-safe via `Buffer`. Correto.
- **`composeSystemPrompt`:** `advancedOverride` curto-circuita; `biSchema` só é
  anexado quando passado, e `runAgent` só o passa para `admin`/`super_admin`
  (`BI_ROLES`). Caminho 3c gated corretamente.
