## F3.5c — Tabela de dados profissional

**Objetivo:** a `DataTable` ganha as interações da `conversas-table` do
nexus-insights. Componente genérico (a F6 reusa).

**Porte de:** `nexus-insights/.../conversas-table.tsx`,
`conversa-drill-down.tsx`, `export-button.tsx`.

**Tasks:**
- **c1** — Ordenação multi-coluna: clique cicla asc→desc→remove; Shift+clique
  acumula; indicador com seta + número de prioridade (1, 2, 3…) quando há mais
  de um critério. TDD no reducer de ordenação.
- **c2** — Busca varrendo todas as colunas (texto e número→string), com
  debounce. TDD.
- **c3** — Linhas expansíveis: clique na linha expande um painel de detalhe
  abaixo (chevron, `aria-expanded`, `cursor-pointer`). A `DataTable` recebe uma
  prop `expandDetail(row) => ReactNode | null`.
- **c4** — Detalhe do `saldo-produto`: a query passa a devolver, por produto,
  `detalhePorLocal: { localRotulo, saldo, valor }[]` (nome via
  `limparNomeLocal`); o painel expandido mostra esse detalhamento por armazém.
- **c5** — Exportar CSV: módulo puro `export-csv.ts` (gera CSV pt-BR das linhas
  visíveis — respeita busca/ordenação/colunas) + botão na barra da tabela. TDD
  do gerador de CSV.
- **c6** — Verificação: `tsc`/`eslint`/`jest`/`build`; testes da `DataTable`
  atualizados para cada recurso novo.

## F3.5d — Filtros (simples + avançado + chips)

**Objetivo:** filtragem de verdade, no padrão nexus-insights.

**Porte de:** `filters-dialog.tsx`, `advanced-filters.tsx`,
`applied-filters-chips.tsx`, `quick-filters-popover.tsx`,
`filter-chip-list-popover.tsx`.

**Tasks:**
- **d1** — Dropdowns decentes: os controles de `filter-controls/` abrem para
  baixo, com `max-h` + scroll interno e busca interna; nomes de armazém
  agrupados por empresa. Corrige o menu suspenso que estoura a tela.
- **d2** — Chips de filtros aplicados: abaixo da barra, um chip por filtro
  ativo (rótulo + valor) com "x" para remover e "Limpar todos".
- **d3** — Diálogo de filtros "Simples": acordeão de facetas (armazém,
  família, marca…), cada faceta com busca + checkboxes. Estado na URL.
- **d4** — Diálogo de filtros "Avançado": construtor de condições
  (campo · operador · valor) com conectores E/OU e grupos aninhados. Modelo de
  query serializável (reusável pela F6). TDD do modelo + da serialização.
- **d5** — Verificação completa.

## F3.5e — Presets, atalhos e tour

**Objetivo:** produtividade e onboarding.

**Porte de:** `presets-dialog.tsx`, `presets-popover.tsx`, o tour do
nexus-insights.

**Tasks:**
- **e1** — Presets (buscas salvas) persistidos no banco por usuário: schema
  Prisma + migration + Server Actions (criar/listar/aplicar/excluir/favoritar).
- **e2** — UI de presets: popover para aplicar/salvar o estado atual
  (período + filtros + ordenação + colunas) como preset.
- **e3** — Atalhos de teclado: abrir filtros, focar busca, trocar período,
  exportar — com um painel de ajuda de atalhos.
- **e4** — Tour de onboarding: passos guiados explicando cada área do
  relatório (modal/balão, "Pular", "Próximo", barra de progresso).
- **e5** — Verificação completa.

## F3.5f — Relatórios repensados

**Objetivo:** cada relatório responde uma pergunta de gestor — não é "encher
linguiça". Decisões de produto a confirmar no brainstorm da sub-fase.

**Tasks (a detalhar na spec da sub-fase):**
- **f1** — `valor-armazem`: deixa de ser um gráfico de barras com dezenas de
  barras; vira **lista/tabela** ordenável e filtrável (armazém · valor · nº de
  produtos), com um gráfico-resumo só do top-N. Responsivo no celular.
- **f2** — `entradas-saidas`: além da linha, uma **lista** do que entrou/saiu
  (produto · armazém · quantidade · data), para o relatório significar algo.
- **f3** — `top-movimentados` e `produtos-parados`: revisar a pergunta que
  cada um responde e o detalhamento; KPIs de cabeçalho.
- **f4** — `concentracao`: manter o gráfico mas acrescentar a **lista** por
  trás dele; escala legível.
- **f5** — Verificação completa.

## F3.5g — Frescor do dado (worker)

**Objetivo:** o "Atualizado em" não pode ficar horas atrás.

**Causa diagnosticada:** `src/worker/sync/sync-config.ts` tem
`snapshotIntervalMin: 1440` — o snapshot do estoque roda 1×/dia.

**Tasks:**
- **g1** — Definir o intervalo de snapshot adequado (decisão humana — equilíbrio
  entre frescor e carga na API do Odoo Tauga). Aplicar como novo padrão e/ou
  garantir que a tela `/configuracao` o controla e o worker reage sem restart.
- **g2** — Exibir o frescor de forma honesta e legível ("há 8 min") na UI.
- **g3** — Verificação.

## Ordem e dependências

```
F3.5c (tabela) ──┐
                 ├──► F3.5e (presets dependem de período+filtros+ordenação)
F3.5d (filtros) ─┘
F3.5f (relatórios) — independente, pode correr em paralelo a c/d
F3.5g (worker) — independente, pode ser feito a qualquer momento
```

Sugestão de execução: **F3.5c → F3.5d → F3.5f → F3.5e → F3.5g**.

## Princípios (valem para toda sub-fase)

- Cada sub-fase: spec própria (v1→v3, 2 reviews) + plano próprio (v1→v3,
  2 reviews) + execução TDD em microtarefas + verificação + code/UI review.
- `ui-ux-pro-max` consultado em toda task de UI.
- Componentes genéricos e reutilizáveis — a F6 (construtor de relatórios) vai
  consumir tabela, filtros, presets e export.
- Nada de placeholder; `tsc`/`eslint`/`jest`/`build` verdes ao fim de cada
  sub-fase; commits atômicos.
