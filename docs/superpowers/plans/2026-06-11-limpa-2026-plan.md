# PLAN v3 , Limpa 2026+ (sobre a SPEC v3)

> Spec: docs/superpowers/specs/2026-06-11-limpa-2026-design.md (v3).
> **v3 = v1 + 2 reviews adversariais aplicadas (8 BLOCKERs + 12 MAJOR/MINOR).**
> Execução TDD, 1 unidade por task, commit atômico por task. `--apply` (dev E
> prod) SÓ com aprovação humana do relatório de dry-run.

## Fatos cravados pelas reviews (regras duras da execução)
- **Título (raw):** a chave simples é `data->>'situacao_divida_simples'`
  (`situacao_simples` NÃO existe no raw); a chave `situacao` tem
  `efetivo(4990)/provisorio(4366)/quitado(1815)/baixado(55)`. **`efetivo` =
  dívida VIVA (mapeia para situacao_simples='aberto') , NUNCA deleta.**
  Deleta apenas `situacao_divida_simples IN ('quitado','baixado') AND
  data_pagamento < 2026-01-01` (raw) / `situacaoSimples IN (...)` (fato).
- **FK de filho SPED é many2one array `[id,"label"]`**: id =
  `(data->'documento_id'->>0)::int` , nunca `data->>'documento_id'`.
- **`raw_sped_produto_lote_serie` = 2,9GB (MAIOR tabela; 8.169 linhas, só 11
  pré-2026)**: é quase-mestre (não corta por data), mas o espaço é BLOAT ,
  entra na lista de VACUUM FULL mesmo sem DELETE (o maior ganho real de disco).
- Backfill (`since===null`) é o ramo crítico do filtro (hoje domain `[]`).
- Assinaturas mudam: `syncSnapshot(client, prisma, rawKey, model, corte?)` +
  callers em processors.ts (~125, loop cobre snapshot E estatico);
  `reconcileModel(client, raw, model, corte?)` + caller (~163).
- Pausar worker dev/prod = `docker compose stop worker` (vence o
  `restart: unless-stopped`); religar com `start`. Verificar fila idle antes
  de capturar invariantes.
- Sem migration: constante + AppSetting existente (`sync.corte_dados`,
  só-banco nesta fase).

## Tasks

### T1 , Catálogo `corte` (por CLASSE, 1 commit/classe)
- **T1a** transacionais com data própria: `corte: { odoo, raw, fato? }`.
- **T1b** filhos SPED: `cortePai: { tabela, fkRaw, fkM2O: true }` (extração
  `->0`).
- **T1c** título: `corteEspecial: "titulo_por_situacao"` carregando
  `{ rawSituacao: "situacao_divida_simples", fatoSituacao: "situacaoSimples" }`.
- **T1d** lista negativa nominal (mestres/foto-atual/preço-vigência/lote_serie)
  + teste de CONJUNTO EXATO (snapshot do set de modelos com corte; modelo
  incremental novo sem corte nem lista negativa = teste vermelho).

### T2 , Filtro permanente no sync (3 sub-tasks, 1 por ciclo)
- **T2a incremental**: cláusula em AMBOS os ramos (`since` e `since===null`/
  backfill); título por situação no domínio Odoo. Teste cobre `since===null`.
- **T2b snapshot**: assinatura + callers; cláusula no `searchReadPaged`;
  decidir `estatico` (sem corte , mestres); CR-02: pull MENOR pós-filtro não é
  glitch; **ordem segura**: tabelas snapshot afetadas são cobertas pelo pg_dump
  ANTES do 1º ciclo filtrado (o refresh já "purga" sem rede).
- **T2c reconcile**: assinatura + caller; `searchIds` com a mesma cláusula.
- **T2d reclassificação `estoque.extrato`** (snapshot→incremental+corte): task
  própria com teste de integridade + âncora E2E de estoque.

### T3 , Validação dos campos no Odoo (read-only)
`scripts/limpa/valida-campos-odoo.ts`: `search_count` por modelo com corte
(prova campo filtrável + mede universo 2026). Inclui confirmar a semântica de
`efetivo` contra o Odoo (review #1.2).

### T4 , Purge (4 sub-tasks)
- **T4a** funções puras: predicados por classe (data própria / filho-M2O / 
  título-situação), NULL count, batching (`LIMIT 10000` em loop, commit por
  lote , 221k linhas/923MB não cabem numa transação só). Fixtures com shape
  real `[id,label]`.
- **T4b** dry-run (default, read-only): relatório tabela a tabela (linhas,
  NULLs, `pg_total_relation_size`) salvo em docs/, aprovável.
- **T4c** `--apply`: DELETE em lotes, filhos→pai, log por tabela.
- **T4d** `--vacuum`: VACUUM (FULL, ANALYZE) por tabela tocada + lote_serie
  (bloat 2,9GB) + medição bytes/duração por tabela (a duração de DEV dimensiona
  a janela de PROD).

### T5 , Invariantes financeiros (sequência exata)
1. `docker compose stop worker` + poll fila idle; 2. captura **a_pagar-aberto e
a_receber-aberto SEPARADOS** (nunca soma líquida); 3. apply; 4. rebuild de
`fato_financeiro_titulo` PRIMEIRO + re-captura + comparação (R$ 0,00 cada);
aborta exit!=0 se divergir; 5. demais rebuilds.

### T6 , Rebuild + E2E âncoras 2026
Registry completo (título já validado em T5); âncoras: faturamento mês,
a pagar/receber, estoque (cobre T2d), DFe.

### T7 , Honestidade pré-2026
- **T7a** (executável já, sem LLM): formatadores/período , unit determinístico
  do texto honesto da spec §5.
- **T7b** (PÓS-RECARGA OpenAI): regra de prompt + golden (auditar entradas
  2025; +2 casos de recusa honesta) + smoke bubble.

### T8 , Recalibração 2025 (1 commit por artefato, lista NOMINAL da spec §5)
conferencia-fiscal.ts; gen-baseline-eliminacao.ts + baseline json;
e2e-financeiro-aberto.ts; smoke-e2e.ts; teste-A-venda-shape-completo.py;
f4-baseline.json + f4-baseline.e2e.ts; golden-nex.json + golden-scorecard.json;
golden-under-active.e2e.ts; sync-whitelist.test.ts. Critério de saída: grep
prova zero "2025" hardcoded operante.

### T9 , Execução DEV (ordem)
Pré-flight disco (`df` do volume Postgres: folga >= tabelas a vacuumar + dump;
dump FORA do volume de dados) → pg_dump (transacionais + snapshot afetadas) →
deploy filtro local (worker via `docker compose build app` + recreate) →
1 ciclo verificado → stop worker + fila idle → dry-run → **APROVAÇÃO DO
USUÁRIO** → T5 (captura→apply→invariantes) → T4d vacuum (medir duração) →
T6 → start worker → 2 ciclos verificados.

### T10 , Execução PROD (deploy assistido)
ship.py (T1-T8) → janela comunicada (duração medida em DEV) → mesmo ritual do
T9 no servidor (pré-flight disco do volume Portainer!) → relatório final
commitado + STATUS.

### T11 , Operação/painel

- Revisar painel "Estado da ingestão" (contagens caem de ordem de grandeza ,
  não alarmar).
