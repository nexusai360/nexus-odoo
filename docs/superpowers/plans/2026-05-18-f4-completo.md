## Padrões herdados da onda 1 — obrigatórios em TODO builder/tool novo

Cravados pela SPEC v3 §3 cabeçalho e pela `research §Método`:

- Builders filtram `rawDeleted = false`.
- Dinheiro: `Decimal @db.Decimal(18,2)` no Prisma; no builder `Number(raw.x ?? 0)`.
- Datas `date` do Odoo: `typeof raw.x === "string" ? new Date(\`${raw.x}T00:00:00\`) : null`.
- Campos `selection`: `String` no fato (nunca enum Prisma) — fail-safe SPED.
- Relações m2o via `relId`/`relNome` de `src/worker/fatos/odoo-relational.ts`.
- "Dias de atraso" **não** é coluna — calculado na query da tool. **Função real:
  `diasAtraso(dataVencimento: Date | null, hoje: Date): number` em
  `mcp/lib/dias-atraso.ts`** (achado P-I2 — a v1 citava `calcDiasAtraso`, símbolo
  inexistente; o nome correto é `diasAtraso`).
- Builder registrado em `src/worker/fatos/registry.ts`.
- `atualizadoEm DateTime @default(now())` no schema — o `createMany` **não**
  injeta `atualizadoEm` por linha (decisão N5 da onda 1; o mapper não produz
  `atualizadoEm`).
- **Todos os 6 builders novos são `cycle: "incremental"`** (fontes incrementais
  no `MODEL_CATALOG`) e seguem o padrão transacional: `deleteMany` +
  `createMany` + `markFatoBuilt(tx, ...)` **dentro de um único
  `prisma.$transaction`** — o `tx` é o cliente passado a `markFatoBuilt`
  (`FatoBuildStateClient` aceita `tx`). O **COMMIT é o ponto de consistência**.
  **Sem exceção e sem bifurcação:** os 6 builders, incluindo o de
  `FatoNotaFiscalItem` (211k linhas), seguem este mesmo padrão transacional —
  a SPEC v3 §3.2 (achado N2) é categórica e o plano não a revoga (achado R2-C1).
- **`createMany` sempre envolto em `if (mapped.length) { … }` (achado R2-I2).**
  O `createMany` do Prisma/`@prisma/adapter-pg` lança com array vazio em algumas
  versões; o builder da onda 1 `fato-financeiro-titulo.ts` se protege com
  `if (mapped.length)`. **Regra cravada para os 6 builders novos:** o
  `createMany` (e, no caso de `FatoNotaFiscalItem`, **cada chamada de chunk**)
  é sempre guardado por `if (mapped.length) { … }` / `if (chunk.length) { … }`,
  espelhando `fato-financeiro-titulo.ts`. O mapper-espelho mantém o guard.
- Tool MCP: arquivo próprio em `mcp/tools/<dominio>/<nome>.ts`, exporta uma
  `ToolEntry`, registrada no `mcp/tools/<dominio>/index.ts`. Estrutura espelha
  `mcp/tools/financeiro/contas-a-receber.ts`: `inputSchema`/`outputSchema` Zod,
  `inputSchemaShape`, função `shape()`, handler envolto em `withFreshness`.
- **`inputSchemaShape` é campo obrigatório do `ToolEntry` (achado R2-I5).** O
  tipo `ToolEntry` (`mcp/catalog/types.ts`) exige `inputSchemaShape: ZodRawShape`
  **além** de `inputSchema`. A tool de referência `contas-a-receber.ts` declara
  `inputSchemaShape: inputSchema.shape`. **Regra cravada:** toda `ToolEntry`
  nova declara **ambos** — `inputSchema` (o `z.object({…})`) **e**
  `inputSchemaShape: inputSchema.shape`. Para tools com `inputSchema =
  z.object({})` o `inputSchemaShape` é `inputSchema.shape` (o `.shape` de um
  objeto vazio, `{}`) — declarar explicitamente, não omitir.
- **Regra de array para o "vazio" do `withFreshness` (achado P-M1).**
  `freshness.ts::ARRAY_KEYS_PRIORITY` é `["linhas","titulos","serie","contas",
  "top","familia","marca"]`. Toda tool nova cujo array principal **se chame
  `linhas`** funciona com a detecção automática de "vazio". Toda tool cujo array
  **não** se chame `linhas` (ex.: `etapas`, `filhas`) ou que não tenha array
  (só escalares) **precisa de `isVazio` custom explícito** passado ao
  `withFreshness`. Esta regra é única e vale para todas as ondas B–F; cada task
  abaixo declara qual caso se aplica.

### Extensão do mapa de mocks para os `it()` "viewer com domínio novo" (achado R2-I1)

Os `it()` de "viewer COM o domínio X" exigiriam um perfil que o fixture
**não tem**. **Decisão cravada (alternativa do achado R2-I1):**

- **Ondas B–E:** o `it()` "por perfil" usa **apenas perfis que já existem no
  fixture** — `admin` (vê tudo, via RBAC camada 1 — confirma que as N tools do
  domínio novo estão no catálogo) **e** `viewer` **sem** o domínio (ex.:
  `user-viewer` só `estoque` — confirma que **não** vê as tools do domínio
  novo). Nenhuma onda B–E estende o mapa de mocks.
- **Onda F:** estende o mapa de mocks **uma única vez**. A task **F.4 ganha um
  step explícito** que adiciona, ao mock de `resolveUserContext` em
  `integration.test.ts` **e** ao `PROFILES`/mapa equivalente de `harness.ts`,
  **um usuário `viewer` com um domínio novo** — `user-viewer-comercial` →
  `{ role: "viewer", domains: ["comercial"] }`. Esse perfil serve o teste
  "`viewer` COM `comercial` vê as 5 de comercial; sem ele não vê" e o teste de
  que domínios-vazios aparecem mesmo para esse perfil. Como `comercial` já é um
  domínio do enum desde antes (SPEC §3.5, achado C7), o fixture pode usá-lo.

### Decisão de robustez (P-C2) — igualdade de conjuntos

Cada task de harness, ao atualizar os literais, **também** garante que o `it()`
de "catálogo bruto" e o de "super_admin recebe EXATAMENTE N" comparem o
**conjunto nominal de ids** (igualdade de conjuntos), não só o `length`. Assim
contagem e identidade das tools são verificadas juntas. **R2-M3:** o grep de
literais antigos da V.1 é heurística secundária — o critério PASS/FAIL é a
igualdade de conjuntos de ids, não o grep (ver V.1 Step 6).

### Modelo de execução — paralelismo e sequencialidade (achados P-C3, P-C5)

- **Onda 0:** O.1 → O.2 → O.3 sequenciais (O.3 consolida/confere).
- **Onda A:** A.1 → A.2 → A.3 → A.4 → A.5 **estritamente sequenciais**
  (migrations e schema dependem entre si). **Cravado (P-C3):** A.3
  (`prisma migrate dev`) e A.4 (`psql -f`) operam o **mesmo banco de dev**,
  apontado pela mesma `DATABASE_URL` de dev. A.4 só roda **depois** de A.3 ter
  aplicado a migration nesse banco; A.4 confirma a precondição com
  `\d fato_pedido` antes de rodar o script.
- **Ondas B, C, D, E, F são ESTRITAMENTE SEQUENCIAIS — B→C→D→E→F. Não há
  paralelismo entre ondas (achado P-C5).** As ondas compartilham 4 arquivos de
  escrita (`mcp/catalog/index.ts`, `src/worker/fatos/registry.ts`,
  `mcp/lib/freshness.ts`, `mcp/__tests__/integration.test.ts`); editá-los
  concorrentemente causa conflito de merge ou sobrescrita silenciosa. A regra é
  única: **uma onda só começa quando a anterior fechou a verificação E2E.**
- **Step 0 de toda onda B–H (achado P-I7):** antes de qualquer task da onda,
  garantir `npx prisma generate` executado no checkout atual — o client gerado
  (`src/generated/prisma`) normalmente está em `.gitignore`; sem `prisma
  generate` os tipos `FatoPedido` etc. não existem e os builders/tools não
  compilam. Este Step 0 é parte da verificação de entrada da onda.
- **Precondição de cache das ondas B–E (achado R2-I7).** O E2E de cada onda
  B–E roda os builders contra o cache real e confere contagens fixas. Isso
  pressupõe o cache de dev **populado** para o domínio da onda. **Cravado:** o
  Step 0 de cada onda B–E inclui uma verificação de cache —
  `SELECT count(*) FROM raw_<tabela> WHERE raw_deleted=false;` para a(s)
  fonte(s) da onda (`raw_pedido_documento`/`raw_pedido_parcela` em B;
  `raw_sped_documento`/`raw_sped_documento_item` em C; `raw_res_partner` em D;
  `raw_contabil_conta` em E). **Contagem 0 = erro/bloqueio que chama o humano**
  (o cron de sincronização precisa rodar primeiro) — não é falha de builder e o
  subagente não tenta "consertar".
- **Dentro de cada onda B–E:** o builder vem antes das tools (a tool consulta o
  fato). As tools de um mesmo domínio editam o mesmo arquivo de núcleo de query
  (`comercial.ts`, etc.) e o mesmo `index.ts` de domínio → **sequenciais**.
- **Onda H** depois de A; suas tasks são sequenciais.
- A verificação E2E de cada onda só roda **após** todas as tasks da onda.
