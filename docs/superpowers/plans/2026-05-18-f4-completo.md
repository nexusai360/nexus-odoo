**Goal:** Estender o servidor MCP semântico do nexus-odoo (já em produção desde
a F4 onda 1 — estoque + financeiro, 14 tools) para cobrir **todos os demais
domínios de negócio** do Odoo Tauga (comercial, fiscal, cadastros, contábil) e
tornar o **Caminho 3c (modo BI)** funcional. Ao fim: catálogo de **33 tools**,

(`@modelcontextprotocol/sdk`, Streamable HTTP), registry de catálogo, RBAC 7
camadas, `withFreshness`/`FATO_FONTE`, `McpAuditLog`, registry de builders.

segunda conexão Postgres dedicada ao 3c (role `nexus_mcp_bi`, `pg.Pool` em
módulo próprio `bi-pool.ts`); (b) uma classe de erro `SqlGuardError` em
`mcp/lib/failure.ts` mapeada para `outcome="invalid_input"` (achado P-C1).

**Tech Stack:** TypeScript, `@modelcontextprotocol/sdk`, Prisma 7
(`@prisma/adapter-pg`), Postgres, Zod, Jest (`ts-jest`, transform CJS), `tsx`,
`pg` (driver cru — já dep transitiva via `@prisma/adapter-pg`), **`pgsql-parser`

## Padrões herdados da onda 1 — obrigatórios em TODO builder/tool novo

Cravados pela SPEC v3 §3 cabeçalho e pela `research §Método`:

- Builders filtram `rawDeleted = false`.
- Dinheiro: `Decimal @db.Decimal(18,2)` no Prisma; no builder `Number(raw.x ?? 0)`.
- Datas `date` do Odoo: `typeof raw.x === "string" ? new Date(\`${raw.x}T00:00:00\`) : null`.
- Campos `selection`: `String` no fato (nunca enum Prisma) — fail-safe SPED.
- Relações m2o via `relId`/`relNome` de `src/worker/fatos/odoo-relational.ts`.
- "Dias de atraso" **não** é coluna — calculado na query da tool. **Função real:
  `diasAtraso(dataVencimento: Date | null, hoje: Date): number` em
  `mcp/lib/dias-atraso.ts`** (achado P-I2 — a v1 citava `calcDiasAtraso`, símbolo
  inexistente; o nome correto é `diasAtraso`).
- Builder registrado em `src/worker/fatos/registry.ts`.
- `atualizadoEm DateTime @default(now())` no schema — o `createMany` **não**
  injeta `atualizadoEm` por linha (decisão N5 da onda 1; o mapper não produz
  `atualizadoEm`).
- **Todos os 6 builders novos são `cycle: "incremental"`** (fontes incrementais
  no `MODEL_CATALOG`) e seguem o padrão transacional: `deleteMany` +
  `createMany` + `markFatoBuilt(tx, ...)` **dentro de um único
  `prisma.$transaction`** — o `tx` é o cliente passado a `markFatoBuilt`
  (`FatoBuildStateClient` aceita `tx`). O **COMMIT é o ponto de consistência**.
- Tool MCP: arquivo próprio em `mcp/tools/<dominio>/<nome>.ts`, exporta uma
  `ToolEntry`, registrada no `mcp/tools/<dominio>/index.ts`. Estrutura espelha
  `mcp/tools/financeiro/contas-a-receber.ts`: `inputSchema`/`outputSchema` Zod,
  `inputSchemaShape`, função `shape()`, handler envolto em `withFreshness`.
- **Regra de array para o "vazio" do `withFreshness` (achado P-M1).**
  `freshness.ts::ARRAY_KEYS_PRIORITY` é `["linhas","titulos","serie","contas",
  "top","familia","marca"]`. Toda tool nova cujo array principal **se chame
  `linhas`** funciona com a detecção automática de "vazio". Toda tool cujo array
  **não** se chame `linhas` (ex.: `etapas`, `filhas`) ou que não tenha array
  (só escalares) **precisa de `isVazio` custom explícito** passado ao
  `withFreshness`. Esta regra é única e vale para todas as ondas B–F; cada task
  abaixo declara qual caso se aplica.

### Modelo de execução — paralelismo e sequencialidade (achados P-C3, P-C5)

- **Onda 0:** O.1 → O.2 → O.3 sequenciais (O.3 consolida O.1 e O.2 num doc).
- **Onda A:** A.1 → A.2 → A.3 → A.4 → A.5 **estritamente sequenciais**
  (migrations e schema dependem entre si). **Cravado (P-C3):** A.3
  (`prisma migrate dev`) e A.4 (`psql -f`) operam o **mesmo banco de dev**,
  apontado pela mesma `DATABASE_URL` de dev. A.4 só roda **depois** de A.3 ter
  aplicado a migration nesse banco; A.4 confirma a precondição com
  `\d fato_pedido` antes de rodar o script.
- **Ondas B, C, D, E, F são ESTRITAMENTE SEQUENCIAIS — B→C→D→E→F. Não há
  paralelismo entre ondas (achado P-C5).** A "alternativa paralela" da v1 foi
  **removida**: as ondas compartilham 5 arquivos de escrita
  (`mcp/catalog/index.ts`, `src/worker/fatos/registry.ts`, `mcp/lib/freshness.ts`,
  `prisma/schema.prisma` — não tocado nas ondas B–F, mas listado por
  precaução —, `mcp/__tests__/integration.test.ts`); editá-los concorrentemente
  causa conflito de merge ou sobrescrita silenciosa. A regra é única: **uma onda
  só começa quando a anterior fechou a verificação E2E.**
- **Step 0 de toda onda B–H (achado P-I7):** antes de qualquer task da onda,
  garantir `npx prisma generate` executado no checkout atual — o client gerado
  (`src/generated/prisma`) normalmente está em `.gitignore`; sem `prisma
  generate` os tipos `FatoPedido` etc. não existem e os builders/tools não
  compilam. Este Step 0 é parte da verificação de entrada da onda, registrado
  no relato.
- **Dentro de cada onda B–E:** o builder vem antes das tools (a tool consulta o
  fato). As tools de um mesmo domínio editam o mesmo arquivo de núcleo de query
  (`comercial.ts`, etc.) e o mesmo `index.ts` de domínio → **sequenciais**.
- **Onda H** depois de A; suas tasks são sequenciais (cada uma depende da anterior).
- A verificação E2E de cada onda só roda **após** todas as tasks da onda.
