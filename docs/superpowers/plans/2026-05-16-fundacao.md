# F1 — Fundação — Roteiro de Execução

> **Por que roteiro, e não plano monolítico:** o F1 é uma aplicação inteira.
> Um único plano de 70 tasks seria impossível de double-checar a sério e
> desatualizaria rápido. Aplicando a regra de raiz do `CLAUDE.md` (§6 [4–6] —
> decomposição máxima), o F1 é executado em **6 blocos sequenciais**. Cada
> bloco recebe:
> 1. seu **próprio plano granular** — `docs/superpowers/plans/2026-05-16-fundacao-blocoN-<nome>.md`;
> 2. seu **double-check** (Review #1 + #2);
> 3. sua **verificação** — e só então o bloco seguinte começa.
>
> Isso contém a inconsistência: nada avança sobre base não verificada.

## Blocos

### Bloco 1 — Scaffolding base
Estrutura do projeto Next.js 16, configs, Prisma (esqueleto) e Docker.
Arquivos-alvo: `package.json`, `tsconfig.json`, `next.config.ts`,
`eslint.config.mjs`, `postcss.config.mjs`, `components.json`, `jest.config.ts`,
`.nvmrc`, `prisma/schema.prisma` (esqueleto), `prisma.config.ts`,
`docker/Dockerfile`, `docker/entrypoint.sh`, `docker-compose.yml`,
`.dockerignore`, `.env.local` (dev).
Pronto quando: `npm install` e `npx tsc --noEmit` passam; `docker compose config` valida.

### Bloco 2 — Banco e bibliotecas base
Schema Prisma completo + migration; bibliotecas de `src/lib`.
Arquivos-alvo: `prisma/schema.prisma` (models), `prisma/seed.ts`,
`src/lib/{prisma,pg-pool,redis,encryption,audit,rate-limit}.ts`.
**Atenção:** o `nexus-insights` usa `pg-pool.ts` além do Prisma — portar também.
Pronto quando: migration aplicada, seed roda, `tsc` passa.

### Bloco 3 — Auth e RBAC
NextAuth v5, middleware, helpers de RBAC.
Arquivos-alvo: `src/auth.ts`, `src/auth.config.ts`, `src/middleware.ts`,
`src/lib/auth-helpers.ts`, `src/app/api/auth/[...nextauth]/route.ts`.
**Atenção:** `auth-helpers.ts` tem lógica de Chatwoot (`accountIds`/`teamIds`,
queries em `user_account_access`/`user_team_access`) — remover por completo;
a interface `AuthUser` é reduzida a campos de identidade + papel.
Pronto quando: build passa; teste de hierarquia de papéis passa.

### Bloco 4 — UI base e telas de autenticação
Layout raiz, ThemeProvider, componentes `ui/` base, telas `(auth)`.
Arquivos-alvo: `src/app/layout.tsx`, `globals.css`, `page.tsx`,
`src/components/theme-provider.tsx`, componentes `ui/` necessários (conferir
imports), `(auth)/{layout,login,forgot-password,reset-password}`.
Pronto quando: build passa; login renderiza.

### Bloco 5 — Telas protegidas

Arquivos-alvo: `(protected)/{layout,dashboard,usuarios,perfil}`,
`src/components/users/*`, Server Actions correspondentes, `api/health`,
`api/user/theme`.
**Atenção:** remover de `usuarios` a parte de acesso a contas/times Chatwoot.
Pronto quando: build passa; CRUD de usuários funciona com RBAC.

### Bloco 6 — Worker e CI/CD

## Pontos de atenção globais (achados das reviews do roteiro)

Estes valem para os planos de todos os blocos:

- **`npm install --legacy-peer-deps`** — exigido por Next 16 + NextAuth v5.
- **`.env.local`** — completar com valores de dev no Bloco 1; nunca commitado.
- **`jest.config.ts`** — criado no Bloco 1; sem ele os testes não rodam.
- **`.nvmrc` = 22** — alinhado ao `node:22-alpine` do Dockerfile.
- **`src/generated/`** — gitignored (client Prisma é gerado).
- **Porte de UI** — não trazer os diretórios `__tests__` dos componentes.
- **Verificação** — smoke via `npm run dev`; build Docker validado à parte.
- **Zero Chatwoot/Power BI** — cada bloco confere `grep -ri chatwoot src/`.

## Ordem e estado

Blocos executados em ordem 1→6. O estado de cada bloco (plano / double-check /
execução / verificação) é rastreado por tasks e pelos commits na branch.
