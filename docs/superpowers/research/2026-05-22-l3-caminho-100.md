26 falhas no total. 17 são estruturais (categorias `global` e `notas`); 9 são

## O que é a categoria `global`

`global` são as **5 perguntas de contagem-total** do harness: "quantos serviços
no catálogo?", "quantos parceiros?", "quantos pedidos?", "quantas notas?",
"quantas regras de preço?". É a única categoria com pergunta sobre o total
absoluto de uma tabela.

**Por que falha (4 de 5):** as tools de listagem (`servico_listar`, etc.)
devolvem `linhas` (uma amostra, cortada pelo `limite`) **e** um campo `total`
(a contagem completa). O agente está reportando o tamanho de `linhas` (ex.:
250) em vez de ler o campo `total` (ex.: 336). Na primeira sondagem o agente
respondeu "122 serviços" quando o catálogo tem 336 — mesmo defeito.

## Melhorias para 100%

### A. `global` — contagens-totais (recupera ~4 casos, e blinda o conceito)

O problema é o agente confundir "amostra retornada" com "total". Duas opções,
podem ser combinadas:

1. **Tools de contagem dedicadas.** Criar tools que respondem só um número:
   `servico_contar`, `parceiro_contar` (já existe `cadastro_contar_parceiros`),
   `pedido_contar`, `nota_contar`, `preco_regra_contar`. Cada uma retorna
   `{ total: N }`. Pergunta "quantos X" → tool de contar → resposta inequívoca.
2. **Deixar o `total` explícito** nas tools de listagem: na descrição da tool,
   afirmar "o campo `total` é a contagem completa; `linhas` é só uma amostra
   limitada". E, no shaping da resposta, renomear para algo como
   `totalNoCatalogo` para o modelo não confundir.

Recomendado: opção 1 (tools de contar) — é a que zera a ambiguidade.

### B. `notas_entrada_fornecedor` — filtro por fornecedor (recupera ~13)

O filtro `fornecedor` faz `contains` no nome do participante e a tool agrupa
por `participanteNome`. Quando o nome buscado casa com **mais de um**
participante (matriz/filial, ou nomes parecidos), saem várias linhas e o
agente lê uma só — daí "1 nota" onde o esperado eram 287.

Melhorias:
1. **Modo agregado:** quando `fornecedor` é informado, a tool deve devolver
   também um `totalAgregado` somando todas as linhas que casaram, além das
   linhas individuais. Pergunta "quantas notas do fornecedor X" → o agente lê
   o agregado.
2. **Identificador exato:** aceitar busca por CNPJ/CPF do participante (campo
   já existe no `sped.documento`), que é inequívoco, em vez de só o nome.
3. (Harness) usar o nome completo + CNPJ na pergunta de teste, para o caso ser
   inequívoco — já foi reduzida a ambiguidade, mas não zerada.

### C. Ruído esparso do modelo (9 casos — conta, serviço, preço, estoque, pedidos)

São acertos de ~98-99,5%: o `gpt-5.4-nano` ocasionalmente escolhe o código
errado ou responde "0" mesmo com a tool correta disponível. É erro do modelo,
não da tool. Mitigações:

1. **Descrições de tool mais diretivas** e exemplos no input schema (via
   `.describe()` do Zod, que vai para o JSON schema que o LLM lê).
2. **Few-shot no system prompt** do agente para os padrões que erram (ler o
   campo certo, não inventar código).
3. Avaliar um modelo um pouco maior (`gpt-5.4-mini`) para as perguntas que o
   `nano` erra — o usuário fixou `nano`, então isso é decisão dele.
4. Reexecutar a bateria após A e B para medir o ruído real isolado.

## Estimativa

Resolvendo A e B (estrutural), a assertividade sai de 97,73% para ~99% (só o
ruído esparso de C sobra). Os 100% exatos dependem de C, que tem um piso de
erro do modelo `nano`; provavelmente exige descrições/few-shot afinados e/ou
um modelo maior para o resíduo.

## Próximos passos sugeridos (próxima sessão)

1. Implementar as tools de contagem (A) — rápido, alto impacto.
2. Adicionar `totalAgregado` e busca por CNPJ na tool de notas (B).
3. Afinar descrições de tool com `.describe()` e few-shot (C).
4. Reexecutar `f4l-l3-harness.ts` (com autorização — custo OpenAI) e medir.
